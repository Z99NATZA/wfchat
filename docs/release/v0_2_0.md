# v0.2.0 - Automatic Memory

Release date: 2026-07-12

Automatic-memory release replacing the retired manual facts and summaries
system. Aiko can now learn selected durable user context after successful chat
turns and reuse relevant memories naturally in later conversations, including
across chats and between Thai and English topic wording.

## Highlights

- Added durable, selective automatic memory capture with message-level
  provenance.
- Added bounded cross-chat retrieval for the exact owner and character.
- Added Thai-English canonical topic matching without embeddings or a vector
  database.
- Added one shared learned-context preparation path for streaming and
  non-streaming chat requests.
- Added a confirmed Settings action that resets learned context while retaining
  chat history.
- Added deterministic evaluation, lifecycle safety tests, expiration coverage,
  and privacy-safe operational telemetry.

## Automatic Capture

- Enqueues one idempotent extraction job in the same transaction as each
  successfully persisted user/assistant turn.
- Processes extraction outside the response-critical path through a durable
  PostgreSQL outbox worker.
- Reclaims stale jobs, applies bounded retry backoff, and moves a third failed
  attempt to `dead` without rolling back a successful chat response.
- Uses a strict structured schema with at most five memory candidates.
- Requires candidate evidence to be grounded in an exact substring of the
  persisted user message.
- Rejects unsupported inferences, temporary instructions, low-value details,
  secrets, credentials, financial identifiers, and unsafe candidate values.
- Reinforces repeated evidence and replaces an existing value only when the
  user explicitly provides corrective evidence.

## Retrieval And Prompt Context

- Prefilters at most 50 candidates by exact guest/account ownership,
  `character_id`, supported memory kind, minimum `0.65` confidence, expiration,
  and structured topic relevance.
- Ranks candidates deterministically using relevance, confidence, importance,
  source reinforcement, and recency.
- Injects at most five selected items within 1,200 Unicode characters and an
  estimated 300-token budget.
- Places `LEARNED_CONTEXT_V1` after the character prompt and before current-chat
  history for both regular and SSE streaming requests.
- Treats learned context as untrusted soft guidance, prefers the latest user
  message on conflict, and qualifies uncertain recollections.
- Fails open when memory retrieval encounters a database-specific error, so
  normal chat can continue without learned context.

### Multilingual Topic Matching

- Added a bounded backend-owned canonical taxonomy for `music`, `gaming`,
  `food`, `travel`, `anime`, and `coding`.
- Expands known Thai and English aliases, allowing a Thai message about `เพลง`
  to retrieve an English-tagged `music`/`nightcore` preference and vice versa.
- Uses the same expanded signals for PostgreSQL candidate selection and
  application scoring.
- Prioritizes specific terms such as `nightcore` over broad category-only
  matches.
- Limits broad category-only results so a general topic does not flood the
  prompt with weakly related memories.
- Supports legacy tag aliases during retrieval without a data backfill.

## Ownership And Lifecycle

- Isolates guest memory by session and registered memory by account.
- Shares registered-account memory across the account's browser sessions while
  retaining strict character isolation.
- Merges duplicate keys and preserves distinct sources when a guest session is
  promoted to a registered account.
- Tracks one memory across multiple source chats so repeated evidence
  reinforces rather than duplicates it.
- Deletes an orphaned memory when its only source chat or source message is
  removed.
- Retains and recalculates memory when another valid source remains.
- Excludes memories when `expires_at <= now()` without requiring a background
  expiration service.
- Prevents pending, retrying, processing, or stale claimed extraction jobs from
  recreating memory after a learned-context reset.

## Privacy And User Control

- Replaced the retired manual facts/summaries manager with automatic selective
  capture; there is no memory list, per-item editor, or browser sync cache for
  learned context.
- Added `DELETE /api/learned-context` for the current owner.
- Added one destructive Settings button with confirmation, using the configured
  character name in localized English and Thai text.
- Reset removes memory items, sources, and queued extraction jobs while keeping
  chat history.
- Operational logs and in-process counters contain only aggregate counts,
  bounded sizes, attempts, and sanitized error codes—not user messages, memory
  content, prompt blocks, credentials, or owner/chat/job identifiers.

## Validation And Operations

- Added a deterministic provider-free automatic-memory evaluation suite with
  English, Thai, and cross-language fixtures.
- Added PostgreSQL coverage for guest/account/character isolation,
  reinforcement, correction, expiration, source deletion, account promotion,
  reset, and extraction job states.
- Added regression coverage for specific-versus-broad ranking, legacy tags,
  strict prompt budgets, privacy filtering, and streaming/non-streaming parity.
- Automatic-memory migrations remain embedded in the API startup path and work
  with the normal Docker Compose deployment.

## Known Limitations

- Capture is selective and asynchronous; not every conversation detail becomes
  memory or becomes available immediately after a turn.
- Multilingual retrieval uses a bounded deterministic topic taxonomy rather
  than general semantic embeddings, so highly indirect wording may not match.
- Learned context is not shared between different characters.
- Per-memory viewing, editing, and deletion are intentionally not exposed.

## Upgrade Notes

- Back up PostgreSQL before upgrading, as with any release that changes stored
  data.
- The API applies pending embedded migrations automatically during startup.
- The retired manual memory tables and manual-memory sync entities are removed;
  old manual facts and summaries are not migrated into automatic memories.
- Existing chat history is retained. New automatic memories are learned only
  from successful turns processed by the extraction pipeline after upgrade.
- No vector database, embedding service, or additional infrastructure is
  required.
- No multilingual memory backfill is required; existing supported legacy tags
  are normalized during retrieval.
- `AI_PROVIDER=mock` completes extraction jobs without creating memory
  candidates. OpenAI-compatible `openai`, `xai`, and `lmstudio` configurations
  use their existing backend model/base URL/credential settings for capture.
